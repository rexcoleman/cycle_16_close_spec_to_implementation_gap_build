## 0. Status, authority, and the freeze-before-count contract

This is the **canonical frozen spec-class definition** for the cycles-1-15 spec denominator. It is **derived top-down** from the four canonical class semantics already in `docs/spec_registry_schema.ttl` `cycle16:SpecType` and `HYPOTHESIS_REGISTRY.md` §3c + H2 (CONFIRMED). It does **not** invent new classes and does **not** replace those definitions — it **sharpens them into a method-stable membership predicate + a single counting unit** so that all three enumeration methods (M1 recorded-registry scan, M2 authored-intent scan, M3 identifier-census) count **the same objects at the same grain**.

**Freeze-before-count contract (the load-bearing order — substrate §3.1).** This artifact is committed FIRST, as a standalone canonical doc, BEFORE any re-count runs. The three enumeration methods then each apply THIS frozen definition independently. **Agreement across the three methods is MEASURED, never engineered.** The Build-Runner's re-run (deliverable (b), next dispatch) MUST cite this artifact by path + commit SHA, and that citation MUST point to a commit that is an ancestor of the re-run commit (the freeze precedes the count; verify via `git merge-base --is-ancestor <frozen-def-commit> <rerun-commit>`). A re-run that does not cite a prior-committed frozen definition is REFUSED.

**Why this artifact exists (the recursive-failure framing).** The S21 HOLD found the denominator has no method-stable operational definition: three enumerations of "distinct cycles-1-15 spec" gave M1 = 232 [measured] / dual-method-union = 354 [measured] / M3 identifier-census = 898 [measured] — a ~3.5× spread. That is the operational-definition substitution one level up (`feedback_operational_definition_substitution.md`): the cycle's own thesis recurring at its own denominator. Drawing the class boundary around whatever makes 232/354/898 collapse to agreement is the substitution this artifact is built to refuse. The definition is drawn from class SEMANTICS; whatever agreement results is a finding, not a target.

## 1. The counting UNIT (method-stable) — one canonical-identity tuple per distinct spec

> **One distinct spec = one (spec_class, canonical_identity) pair, where `canonical_identity` is the realpath-normalized canonical artifact location of the spec's single authoritative definition, deduplicated by the SHA-256 of the tuple `(repo, realpath(canonical_artifact_path), spec_class)`.** The counting object is the **authoritative spec definition site**, NOT an identifier occurrence and NOT a registry row. Three methods count the SAME objects iff they each resolve every candidate to this tuple and dedup by its hash before counting.

**Why this unit, and why the 232/354/898 spread is largely a grain mismatch [heuristic, from method descriptions].** M1 counted minted **registry rows** keyed on `sha256(audit_tuple)` where `audit_tuple = (repo, realpath, artifact_type)` — i.e. already at the authoritative-definition-site grain (232 distinct after idempotent minting, from 268 raw rows / 36 collisions). M3's identifier-census counted **identifier occurrences** across 15 repos (898) — every `DP#N`/`Done #N`/`BE-X` *mention*, which inflates ~3.5× because one spec is cited many times and because references (not specs) are counted. M2's dual-method-union (354) sits between because it adds authored-intent tokens to M1 without first collapsing them to the definition-site grain. **Fixing the unit to "one authoritative definition site per (class, canonical-location)" removes the occurrence-vs-site grain mismatch that is the dominant driver of the spread.** Residual divergence after the unit is fixed is an honest finding (see §5), NOT something to tune away.

**Canonical-identity resolution rule (decidable by a scanner).** For each candidate spec artifact:
1. Resolve its canonical definition location: AgentContract → the `.claude/agents/<name>.md` file realpath; Schema → the schema-definition file realpath (`.ttl` shape / pydantic model file / JSON-Schema file) where the schema is *defined* (not where it is referenced); DesignDecision → the `DECISION_LOG.md` / `state.json` structured-row anchor where the decision is *recorded as a decision* (the ADR/disposition row, not a downstream citation of it); MethodologyCommitment → the cycle-close `FINDINGS.md` Layer-5 anchor where the commitment is *stated* (not a downstream warmup citation of it).
2. Compute `canonical_identity = sha256(f"{repo}|{realpath(canonical_location)}|{spec_class}")[:N]`.
3. Two candidates with the same `canonical_identity` are the SAME spec (collapse). This is exactly M1's existing `spec_id_minting` rule generalized so M2 and M3 can apply it identically.

## 2. Per-class membership predicate (necessary + sufficient; derived top-down from §canonical semantics)

For each of the four classes, an artifact is a **spec of this class** iff it satisfies the predicate below. Each predicate is decidable by a scanner (the Build-Runner encodes it); each is derived from the class's canonical `rdfs:comment` / H2 verifier, NOT from a target count.

| Class | Membership predicate (necessary + sufficient) | Derivation (from class semantics) |
|---|---|---|
| **AgentContract** | A file exists at the canonical path `<repo>/.claude/agents/<name>.md` whose body declares an agent role/contract (prompt spec). The spec IS the file; one file = one AgentContract. | The schema `rdfs:comment` defines an AgentContract AS "Agent prompt spec at canonical `.claude/agents/<name>.md`." The *implemented* test (role invoked ≥1 session) is downstream verification, not membership — so membership = the canonical file exists at the canonical path. Drawn at the file because the file is the single authoritative definition site. |
| **Schema** | A schema is **defined** (a JSON Schema doc OR a pydantic model OR a dataclass OR a `.ttl`/SHACL shape) at a definition site, paired with validation code that references it. One schema *definition* = one Schema spec (not one validated instance, not one validation event). | The `rdfs:comment` defines a Schema AS "Schema spec (JSON Schema OR pydantic model OR dataclass) + validation code." The *implemented* test (validation fires at runtime) is downstream. Membership is drawn at the schema-DEFINITION grain (per HR §3c KT-4 note: per-schema-definition, NOT per-instance event class) so two runtime instances of one schema do not double-count. |
| **DesignDecision** | A design decision (ADR-class) is **recorded as a decision** in a `DECISION_LOG.md` ADR row OR a `state.json` `paradigm_dispositions`/`decisions_log`/`honest_carries` structured block. One recorded decision row = one DesignDecision spec. A downstream *citation* of a decision is NOT a new DesignDecision. | The `rdfs:comment` defines a DesignDecision AS "Design decision spec (ADR-class)"; the *implemented* test is "code embodies the decision OR explicit retraction ADR." Membership = the decision is recorded at its ADR/disposition site. Drawn at the recorded-decision row because that is where a decision is authoritatively *made*, distinguishing a decision from the (many) places it is later referenced. |
| **MethodologyCommitment** | A Layer-5 methodology commitment is **stated** in a cycle-close `FINDINGS.md` Layer-5 block (an HC/DP/PL/GPL/MR/MC/LL/R/Pattern/Binding/Check/Discipline/Trap commitment authored AS a methodology commitment for forward cycles). One stated commitment at its authoring FINDINGS Layer-5 site = one MethodologyCommitment spec. A downstream warmup *citation* is NOT a new commitment. | The `rdfs:comment` defines it AS "Layer-5 methodology commitment spec"; the *implemented* test is "documented in cycle-close FINDINGS Layer-5 + cited downstream OR explicit deferral." Membership = the commitment is *stated* at its authoring Layer-5 site. Drawn at the statement site (not the citation) so the DP#26 citation-based-activation surface counts the commitment once, where authored, not once per downstream citation. |

**Decidability note (honest gap, §5):** The DesignDecision and MethodologyCommitment predicates require distinguishing a *recorded/stated decision* from a *citation/reference* to one. That boundary is partially judgment-dependent at the margins (a FINDINGS line may both state and cite). The frozen rule is: **the authoritative definition SITE governs** — the artifact counts at the one location where it is authored as a spec, never at any location that references it. This rule is decidable for the unambiguous majority; the irreducibly ambiguous margin is disclosed in §5, NOT resolved by choosing whichever side hits a target count.

## 3. Explicit exclusions (the contamination fix — substrate §4.3 + S21 HOLD finding 2)

The S21 "122 authored_but_unrecorded" was contaminated: 69 distinct tokens dominated by **46 `DP#N` design-principle references** + **54 Cycle-16-own `Done #N` items** + ~8 naming-mismatch false positives. None of those is a cycles-1-15 spec. The frozen definition excludes them:

1. **Principle-reference tokens are NOT specs (excluded from the denominator).** `DP#N` (design principle), `HC #N` / `HC#N` (hard constraint), `GPL-N` (governance principle), `Pattern N`, `Binding N` tokens are **references to standing principles, NOT spec class members**. A spec MAY cite a `DP#N` (citation ≠ membership); the citation does not mint a denominator member. **Why:** none of `DP/HC/GPL/Pattern/Binding` is one of the four `cycle16:SpecType` classes — they are the standing-principle layer the specs are governed BY, structurally distinct from the spec layer. This is the LOCKED scope (substrate §1): the denominator population is **the four RQS spec classes ONLY**; this is the RQS's own answer, not a Rex question.
2. **Cycle-16-own `Done #N` items are NOT cycles-1-15 specs (excluded from the cycles-1-15 denominator).** The `Done #N` items are **Cycle 16's own recovery scope** (the current cycle's completion bar). They are excluded from the cycles-1-15 audited population by the scope-boundary rule (§4): `cycle_authored == 16` ⇒ out of the cycles-1-15 denominator. **Why:** the denominator audits what cycles 1-15 *intended*; Cycle 16's own recovery commitments are the auditing instrument, not the audited population — counting them is the instrument measuring itself.
3. **Naming-mismatch / token-substring false positives are removed by canonical-identity dedup.** The ~8 false positives arose from token-substring matching (e.g. an authored marker matching a substring of an unrelated recorded name). **Dedup/canonicalization rule:** a candidate is admitted ONLY if it resolves to a canonical definition SITE per §1 (a real file/row/anchor), not merely a token-substring co-occurrence. The absence-of-record test is re-grounded from "token-substring present in any recorded name" to "no recorded canonical-identity tuple matches this candidate's resolved canonical-identity tuple." **Why:** substring co-occurrence is not identity; resolving to a definition site and deduping by the canonical-identity hash removes the substring false-positives structurally.

## 4. Scope boundary — cycles 1-15 only (Cycle-16 authorship excluded)

> **The audited population is specs whose `cycle_authored ∈ {1,…,15}`.** A spec's authoring cycle is determined from its canonical definition site: the `cycle_authored` field on the M1 record (resolved from the repo/path the spec is defined in, e.g. `cycle_NN/...`, and the `session_authored` `Cycle-N-Sm` marker). A candidate with `cycle_authored == 16` (any Cycle-16-authored spec, including all `Done #N` recovery items and any spec first defined in the Cycle-16 repo) is **excluded from the cycles-1-15 denominator**. (Note: the M1 raw population includes 39 Cycle-16-authored records [measured]; these are out-of-scope for the cycles-1-15 denominator and MUST be filtered before the cross-method count.)

**Why drawn at cycles 1-15:** the research question audits the spec→implementation gap of the *prior* (already-authored) cycles; Cycle 16 is the auditing cycle. Including Cycle-16 authorship would mix the instrument with the measured population (the same reason §3.2 excludes Cycle-16 `Done #N`).

## 5. Derivation rationale rollup + honest divergence pre-disclosure (Freeze-before-count, §3.1; ±5%-must-emerge, §3.2)

**Every boundary above is drawn from class semantics, not from a target count.** Summary: the UNIT is the authoritative definition site (because a spec is defined once and cited many times); the four predicates are each the class's canonical `rdfs:comment`/H2 verifier reduced to membership-at-definition-site; the three exclusions each remove a layer that is structurally NOT one of the four classes (principles), NOT the audited population (Cycle-16 recovery), or NOT an identity (substring co-occurrence). No boundary was chosen to move 232/354/898 toward each other.

**±5% must EMERGE, never be tuned-to-agree (substrate §3.2 — REFUSAL authored here).** When the three methods (M1 recorded-registry scan, M2 authored-intent scan, M3 identifier-census) are each re-run under THIS frozen definition and the same counting unit, the cross-method spread is **MEASURED**. If the methods agree within ±5%, that agreement is reported as an emergent finding. **If, under this frozen principled definition, the three methods STILL diverge beyond ±5%, that divergence is an HONEST FINDING and is disclosed — the Build-Runner MUST NOT loosen, redraw, or re-tune this definition to force the numbers together.** A stable-because-tuned number is the operational-definition substitution recurring one level up and is REFUSED. On residual divergence, proceed with the most-defensible single denominator (M1's definition-site count under this frozen unit is the most-defensible: it counts authoritative definition sites directly) stated as **"100% of the discoverable population, residual R disclosed"** (denominator-honesty). **The residual R is disclosed, never zeroed, never estimated to zero.**

**Pre-disclosed reasons the three methods MAY still not reach ±5% under this frozen definition [honest pre-disclosure, anecdotal until measured]:** (i) M3's identifier-census reads 15 repos by identifier occurrence; even after collapsing to definition sites, repos may contain definition sites M1's recorded scan never recorded (genuine recorded-vs-authored divergence, which is the #50 finding, not an error). (ii) The DesignDecision/MethodologyCommitment statement-vs-citation boundary (§2 decidability note) has an irreducibly judgment-dependent margin; different methods may resolve a handful of margin cases differently. (iii) M2's authored-intent stream (git/prose/transcript) discovers definition sites never recorded by M1 — the `authored_but_unrecorded` set — which by design is non-zero and is the denominator-honesty finding itself. **Any of these producing >±5% residual is a legitimate disclosed finding, not a failure to engineer away.**
